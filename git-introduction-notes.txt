Git学习笔记（入门篇）

/*
核心篇
**/
一、Git是干什么的
//Git是一个版本管理工具
核心作用：
1.记录项目的历史版本
2.记录每一次改动
3.可以回到以前的版本
4.可以查看改了什么
5.可以管理多条开发线（分支）


二、配置用户名和邮箱
//Git需要知道这次提交是谁做的
查看是否已配置：
git config --global user.name
git config --global user.email

设置用户名和邮箱：
git config --global user.name "your name"
git config --global user.email "your email"

查看全局配置：
git config --global --list

补充说明：
1. --global 表示全局配置，对当前电脑上的所有 Git 仓库生效
2. 如果不加 --global，就是只对当前仓库生效


三、创建Git仓库
在当前目录下初始化一个Git仓库：
git init

补充说明：
1. 
.git 是 Git 仓库的隐藏目录
2. 
里面保存了历史版本、分支信息、提交记录等 Git 管理数据
3. 
有 .git 的目录，通常可以认为是一个 Git 仓库


四、查看当前仓库状态
git status
常见输出解释：
1. On branch main
当前在 main 分支上
2. No commits yet
这个仓库还没有任何提交记录
3. nothing to commit, working tree clean
当前文件状态和最近一次提交一致，没有新的改动
4. Untracked files:
Git 看到了这些文件，但还没有正式跟踪它们
5. Changes not staged for commit:
文件已经修改了，但这些修改还没有进入暂存区
6. Changes to be committed:
这些修改已经进入暂存区，等待提交


五、Git的三个区域
1.工作区
当前真实看到和修改的文件
2.暂存区
执行 git add 后，准备提交的内容放在这里
3.提交历史
执行 git commit 后，真正保存下来的版本历史

改文件 → 放进暂存区 → 正式提交到历史里


六、把文件的当前状态加入暂存区
例如：
git add a.txt

作用：
把 a.txt 当前的状态加入暂存区，准备下次提交

说明：
它不是最终保存
它只是表示：下次提交时，我想把这个版本的文件提交进去
文件以后再次修改后，如果想把新的修改也提交进去，通常还要再执行一次 git add

常见写法：
git add 文件名
git add .

补充：
git add . 表示把当前目录下的改动统一加入暂存区


七、把暂存区里的内容正式保存为一次提交记录
例如：
git commit -m "第一次提交"

解释：
git commit：创建提交
-m：直接写提交说明
"第一次提交"：本次提交的说明文字

常见输出解释：
1.
[main (root-commit) 62b5733] 第一次提交
含义：
main：当前分支
(root-commit)：这是仓库的第一次提交
62b5733：这次提交的短编号
第一次提交：提交说明
2.
1 file changed, 0 insertions(+), 0 deletions(-)
表示这次提交影响了 1 个文件，增加/删除了多少行
3.
create mode 100644 a.txt
表示 Git 在这次提交中正式记录了这个新文件


八、一个最常用的基本流程
git status
git add 文件名
git commit -m "提交说明"
git log --oneline

可以理解为：
先看状态
再把要提交的内容放进暂存区
再正式提交
最后查看历史记录


九、查看提交历史
查看完整历史：
git log

常见输出解释：
commit 62b573343bbe82c8403f9e8a42abc80d6f13ca03 (HEAD -> main)
Author: QingtengW <2072522871@qq.com>
Date:   Sat Apr 4 16:09:23 2026 +0800

    第一次提交

含义：
1. commit ...：提交的完整编号
2. HEAD -> main：当前所在位置就是 main 分支的最新提交
3. Author：提交作者
4. Date：提交时间
5. 最下面缩进的一行：提交说明

更简洁的历史查看：
git log --oneline

更适合看分支结构的历史：
git log --oneline --graph --all --decorate

补充：
当执行 git log 时，Git 可能会进入分页查看界面，底部显示 (END)，表示已经看到最后一页
退出方法：
q


十、查看修改了什么
查看工作区和暂存区之间的差异：
git diff

查看暂存区和最近一次提交之间的差异：
git diff --staged

git diff 最重要的三条规则：
1. - 开头：旧内容
2. + 开头：新内容
3. 没有符号：上下文，表示这一行没变

例：
@@ -1,2 +1,3 @@
好好学习，天天向上
-华风夏韵，洛水天依
+华风夏韵，洛水天依
+永远喜欢洛天依

意思是：
1. 第一行没变
2. 原来某处是一行
3. 现在变成了两行
4. 本质上是增加了 永远喜欢洛天依

十一、版本回退
1. 临时查看旧版本
git checkout 提交号
或者：
git switch --detach 提交号
特点：
1). 文件内容会变成那个提交对应的样子
2). 这不是正式让 main 永久回退
3). 会进入 detached HEAD 状态

2. 真正把当前分支退回旧版本
git reset --hard 提交号
特点：
1). 当前文件会直接变成旧版本的样子
2). 当前分支也会回到旧位置
3). 当前未提交改动会被清掉
注意：
这是一个比较危险的命令，未提交内容可能直接丢失

3. 追加一次提交来撤销旧提交
git revert 提交号
特点：
1). 不会强行回退历史
2). 而是新增一次“反向修改”的提交
3). 更适合已经共享出去的提交记录

十二、detached HEAD 是什么
HEAD 表示当前所在的位置

执行：
git checkout 某个提交号
或者：
git switch --detach 某个提交号

这时不是切到某个分支，而是直接切到某个具体提交
这就叫 detached HEAD（分离头指针）

1. detached HEAD 下提交会怎样？
会长出新的提交
但这些提交不一定挂在某个正式分支上
容易形成“游离的一条线”

2. detached HEAD 的风险
暂时能看到
但没有分支名指着它们
以后切回别的分支后，更容易找不到这条线

3. 如何回到正常状态
git checkout main
或者：
git switch main

4. 如果我想保住 detached HEAD 下的新提交怎么办？
可以立刻新建一个分支接住它：
git branch 新分支名


十三、告诉 Git哪些文件不要纳入版本管理
.gitignore 通常放在项目根目录中

创建空文件：
touch .gitignore

打开文件：
code .gitignore

通常有几类文件不应该提交：
1. 本地配置文件
2. 编译生成文件
3. 临时文件 / 缓存文件
4. 敏感信息
5. IDE 自动生成的无关文件

.gitignore 文件写法：
一行写一条规则

例如：
*.o
*.exe
build/
*.log
.vscode/

补充说明：
1.
.gitignore 主要对还没有被 Git 跟踪的文件生效
2.
如果某文件已经提交过，再写进 .gitignore，Git 也不会自动忽略它
3.
这种情况通常要先把它从 Git 跟踪中移除，再让 .gitignore 生效
例如：
git rm --cached 文件名


十四、删除文件
删除文件并让 Git 记录这次删除：
git rm 文件名

如果你是手动把文件删掉的，也可以再执行：
git add .
这样 Git 会把“删除文件”这件事也加入暂存区


/*
分支篇
**/
一、分支是干什么的
//把工作从主线上分离出去
分支的作用：
1. 让不同功能开发互不影响
2. 可以多人并行开发
3. 主线保持稳定
4. 功能开发完成后再合并回来

可以简单理解为：
分支就是一条独立开发线

二、查看本地分支
git branch

说明：
1. 会列出当前仓库里的本地分支
2. 前面带 * 的分支，就是你当前所在的分支


三、创建本地分支
git branch 分支名

说明：
1. 创建分支的那一刻，新分支和当前分支会先指向同一个提交
2. 这时候它们只是“名字不同”，位置还一样
3. 只有后续继续提交，两个分支才会真正分开


四、切换分支
git checkout 分支名
或者：
git switch 分支名

说明：
切换分支后，工作区文件内容会尽量变成该分支对应的样子

补充：
现在更推荐用：git switch 分支名


五、创建并切换到新分支
老写法：
git checkout -b 分支名

新写法：
git switch -c 分支名

作用：
创建分支 + 立刻切换过去


六、合并分支
基本流程：
1. 先切换到你想接收内容的分支
2. 再执行 git merge 分支名

例如：
git checkout main
git merge dev

含义：
把 dev 分支的内容合并到当前分支 main

注意：
git merge 分支名 的意思不是“切到那个分支去合并”
而是：把指定分支合并到当前分支


七、删除分支
git branch -d 分支名
git branch -D 分支名
区别：
1.
-d：安全删除
只有当分支内容已经被合并等情况下，Git 才会删除
2.
-D：强制删除
不管有没有合并，直接删

注意：
删除的只是分支名，本质上是删掉那根“指针”
不是立刻把所有提交都彻底抹掉


八、合并分支冲突
常见冲突场景：
1. 改了同一个文件的同一位置
2. 一个分支删了某行，另一个分支改了那一行
3. 一个分支删了某个文件，另一个分支还在改这个文件

冲突文件的内容：
/*
*   <<<<<<< HEAD
*   当前分支内容
*   =======
*   要合并进来的分支内容
*   >>>>>>> 分支名
**/

解决冲突的办法：
1. 手动编辑文件
2. 决定最后保留什么内容
3. 删掉冲突标记
4. 再加入暂存区并提交

解决冲突后的流程：
git add 文件名
git commit -m "resolve merge conflict"


九、多个来源分支合并到 main 怎么理解
例如：
main 有两个来源分支：feature-login 和 feature-pay

这并不奇怪，因为 main 本来就可以接收多个分支的合并结果

例如：
git checkout main
git merge feature-login
git merge feature-pay

含义：
1. 先把登录功能合并到 main
2. 再把支付功能合并到 main
3. 最后 main 同时拥有两个功能的成果

所以：
一个主分支有多个来源分支是很正常的


十、开发中分支使用的原则与流程
master（生产）分支
用于保存线上稳定可发布版本，一般不直接开发，只接受经过测试后的合并。

develop（开发）分支
用于日常开发集成，作为各功能分支合并的主干，保存当前最新开发成果。

feature（功能）分支
从 develop 分出，用于开发单个功能或模块；开发完成后再合并回 develop。

release（发布）分支
从 develop 分出，用于发布前测试、修复小问题，完成后合并到 master 和 develop。

hotfix（热修复）分支
从 master 分出，用于线上紧急问题修复，修复完成后合并回 master 和 develop。


十一、一个常见的开发流程例子
例如要开发“用户登录功能”：
1.先从 develop 拉出功能分支
git switch develop
git switch -c feature-login

2.在 feature-login 上开发并提交
git add .
git commit -m "完成登录功能"

3.开发完成后，切回 develop
git switch develop

4.把功能分支合并回来
git merge feature-login

5.如果确认没用了，再删除功能分支
git branch -d feature-login


/*
总结篇
**/
一、Git学习中最容易混淆的几件事
1. git add 不是提交
git add 只是把内容放进暂存区
真正保存历史的是 git commit

2. 创建分支不会自动分开
创建分支那一刻，两个分支通常还指向同一个提交
只有后续继续提交，才会慢慢分开

3. merge 是把“别的分支”合并到“当前分支”
一定要先看清自己现在站在哪个分支上

4. .gitignore 不是万能隐藏术
它主要忽略还没被 Git 跟踪的文件
已经提交过的文件，不会因为写进 .gitignore 就自动消失

5. reset --hard 有风险
它可能直接清掉未提交改动
用之前一定要非常小心
