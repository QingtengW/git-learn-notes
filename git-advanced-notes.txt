Git学习笔记（进阶篇）

/*
远程仓库篇（HTTP/HTTPS协议）
**/
本地仓库：自己电脑里的 Git 仓库
远程仓库：网上的 Git 仓库

远程仓库的作用：
1. 备份代码
2. 多设备同步
3. 团队协作
4. 展示项目

一、创建线上仓库
1.1 登录 GitHub 账号
https://github.com

1.2 在页面右上角点击 + ，选择 New repository（新建仓库）

1.3 填写仓库信息
Repository name（仓库名）
名称通常只能包含字母、数字、连字符（-）等，尽量不要写空格

Description（描述，可选）

Visibility（可见性）
Public：公开
Private：私有

1.4 点击 Create repository（创建仓库）

注意：
如果你本地已经有 Git 仓库和提交记录了，
创建远程仓库时，最好不要勾选：
1) Initialize this repository with a README
2) Add .gitignore
3) Choose a license

原因：
避免本地和远程初始历史不一致，减少后续冲突


二、从本地上传现有项目
适用场景：
本地已经有项目
本地已经 git init 过
本地已经有提交记录
现在想上传到 GitHub

2.1 链接本地仓库与远程仓库
git remote add origin 远程仓库地址

解释：
1) remote
表示远程仓库
2) origin
这是远程仓库的默认名字
可以理解成：
我给这个远程仓库起了个名字，叫 origin
例如：
git remote add origin https://github.com/QingtengW/git-learn-notes.git

补充：
1) 修改远程仓库名字
git remote rename 旧名字 新名字
2) 修改远程仓库地址
git remote set-url 远程仓库名 新的远程仓库地址


2.2 检查是否连接成功
git remote -v

如果成功，你会看到类似：

origin  https://github.com/QingtengW/git-learn-notes.git (fetch)
origin  https://github.com/QingtengW/git-learn-notes.git (push)

含义：
fetch：拉取用的地址
push：推送用的地址

2.3 把本地内容第一次推送到远程仓库
git push -u origin main

解释：
push：把本地提交推到远程
origin：远程仓库名字
main：推到远程的 main 分支
-u：建立“上游跟踪关系”

//可能会弹出来让用户登录GitHub账号的对话框
1) 选 Browser/Device
推荐新手优先使用
一般会跳转浏览器，登录 GitHub 并授权
2) 选 Token 
需要提前生成 Personal Access Token
步骤相对更麻烦，新手一般不优先使用

常见报错：
1) 浏览器认证提示
info: please complete authentication in your browser...
含义：
需要去浏览器完成 GitHub 登录和授权

2) 网络连接失败
例如：
fatal: unable to access 'https://github.com/QingtengW/git-learn-notes.git/':
Recv failure: Connection was reset
或者：
Failed to connect to github.com port 443
Could not connect to server

含义：
本地电脑当前没有成功连接到 GitHub
这通常是网络问题，不一定是 Git 命令写错了

常见排查方向：
1) 检查浏览器能不能打开 GitHub
2) 检查代理是否有问题

代理问题的处理方法（重要）：
打开Git CMD，添加全局代理
git config --global http.proxy http://127.0.0.1:7897
git config --global https.proxy http://127.0.0.1:7897
验证代理设置
git config --global --get http.proxy
git config --global --get https.proxy
取消代理
git config --global --unset http.proxy
git config --global --unset https.proxy

2.4 总结：
git remote add origin 远程仓库地址
git remote -v
git push -u origin main
第一次推送成功后，后面最常用：
git push


三、克隆线上仓库到本地
适用场景：
GitHub 上已经有一个仓库
本地还没有这个项目
想把远程仓库下载到自己电脑
想换一台电脑继续开发
想下载别人的公开项目学习

3.1 clone 是什么意思
命令：
git clone 远程仓库地址

作用：
把远程仓库下载到本地
自动生成一个项目文件夹
自动把远程仓库和本地仓库连接起来
自动把远程仓库默认命名为 origin

所以 clone 比 download zip 强很多，
因为它下载的不只是文件，还有 Git 历史记录。

3.2 和直接下载 ZIP 的区别
下载 ZIP
只是把当前文件下载下来
通常没有 Git 历史
不能直接继续 Git 版本管理流程

git clone
下载的是一个完整 Git 仓库
包括：
项目文件
提交历史
分支信息
远程仓库连接信息

3.3 基本流程
第一步：找到远程仓库地址
去 GitHub 仓库页面，点击：
Code
会看到仓库地址，通常有两种：
HTTPS 方式
https://github.com/用户名/仓库名.git
SSH 方式
git@github.com:用户名/仓库名.git
例如：
https://github.com/QingtengW/git-learn-notes.git

第二步：进入你想保存项目的目录
例如你想把仓库下载到 E盘/test 目录下：
cd /e/test
注意：
这里进入的是“父目录”，不是仓库目录本身。
因为 clone 会自动帮你创建一个新文件夹。

第三步：执行 clone
git clone https://github.com/QingtengW/git-learn-notes.git
执行后，Git 会自动：
创建一个文件夹 git-learn-notes
把远程仓库内容下载进去
在里面自动生成 .git
自动把远程仓库记成 origin

第四步：进入克隆下来的仓库
cd git-learn-notes
然后看状态：
git status
通常会看到类似：
On branch main
Your branch is up to date with 'origin/main'.
nothing to commit, working tree clean
说明：
当前在 main 分支
本地已经和远程 origin/main 连接好了
当前没有未提交改动

3.4 常见问题 
q1 clone 后自动帮用户做了什么？
Git 自动帮用户做了这些事：
1) 新建本地项目目录
2) 下载远程仓库内容
3) 下载提交历史
4) 初始化本地 Git 仓库
5) 自动添加远程仓库 origin
6) 默认检出主分支（通常是 main）
所以，clone 下来的项目本身就是 Git 仓库，不需要再 git init。

q2 clone 自己的仓库 与 clone 别人的仓库 的区别？
clone 自己的仓库可以：
下载到本地、修改、提交、再 push 回去
clone 别人的公开仓库可以：
下载、学习、本地修改

q3 如果想换个本地文件夹名字怎么办？
默认情况下：
git clone 仓库地址
会创建和仓库同名的文件夹。

如果想自己指定本地目录名，可以写：
git clone 仓库地址 你想要的本地文件夹名

例如：
git clone https://github.com/QingtengW/git-learn-notes.git my-notes
这样本地创建的文件夹就不是 git-learn-notes，而是 my-notes。

q4 如何把 clone 下来的别人的仓库 push 到自己的仓库
适用场景：
1. 先 clone 了别人的仓库
2. 现在想把这份代码上传到自己的 GitHub 仓库

方法一：添加自己的远程仓库（推荐）
1）先 clone 别人的仓库
git clone 别人的仓库地址
2）进入项目目录
cd 仓库名
3）查看当前远程仓库
git remote -v
默认一般是：
origin  别人的仓库地址 (fetch)
origin  别人的仓库地址 (push)
4）在 GitHub 上新建自己的空仓库
5）把自己的仓库添加为新的远程仓库
git remote add myrepo 自己的仓库地址
例如：
git remote add myrepo https://github.com/你的用户名/你的仓库.git
6）查看远程仓库
git remote -v
这时一般会同时看到：
origin：别人的仓库
myrepo：自己的仓库
7）把本地代码推送到自己的仓库
git push -u myrepo main
说明：
以后如果还想往自己的仓库推送，可以继续用：
git push myrepo main

方法二：直接把 origin 改成自己的仓库
git remote set-url origin 自己的仓库地址
例如：
git remote set-url origin https://github.com/你的用户名/你的仓库.git
然后推送：
git push -u origin main
注意：
这种方法改完后，本地就不再直接记住原来的别人仓库地址了
补充：
更规范的命名方式常常是：
origin：自己的仓库
upstream：原作者仓库


/*
远程仓库篇（SSH协议）
**/
一、SSH 协议是什么

在 Git 连接 GitHub 这类远程仓库时，本质上要解决两个问题：
1) 你怎么找到对方服务器
2) 服务器怎么确认“你是谁”

SSH 就是一种 网络通信协议，全称是：Secure Shell

可以先把它简单理解成：
一种安全地连接远程服务器的方法。

1.1 SSH 在 Git 里到底是干什么的

Git 本身是版本管理工具，当用户执行：git clone / git pull / git push 这些涉及“远程仓库”的操作时，
Git 需要借助某种网络协议去和 GitHub 通信，SSH 就是其中一种通信方式。
也就是说：
Git 负责版本管理，SSH 负责远程连接和身份认证。

1.2 SSH 的核心思想：用“钥匙”证明你是谁

SSH 最大的特点不是“能联网”，
而是它的 身份验证方式 很特别：它不是每次都让你输账号密码，而是使用一对密钥（公钥＆私钥）

公钥：可以公开，放到 GitHub 上
私钥：只能自己保存，留在自己电脑里

当用户用 Git 通过 SSH 连接 GitHub 时，大致过程是：
1) GitHub 先看：这台电脑有没有对应的“私钥”
2) 你本地电脑用私钥完成认证
3) GitHub 再和它保存的公钥进行匹配
4) 匹配成功，就说明你是合法用户

所以 SSH 的本质是：
不是“记住密码”，而是“记住这台电脑的身份”。

1.3 为什么 SSH 配好后会更省事

因为它认证的重点不是“你每次手动登录”，而是“你的电脑已经持有合法私钥”。

所以一旦配置完成，后面你再执行：git clone / git pull / git push ，通常就会顺畅很多，
不用每次都走浏览器授权或重新输入凭证。

可以把 SSH 理解成：
提前给当前电脑办好一张长期可用的通行证。

1.4 SSH 方式适合什么人

1) 经常使用 GitHub 的人
2) 一台电脑会长期开发项目的人
3) 不想每次 push 都登录授权的人
4) 后面会接触多人协作、fork、开源项目的人

对新手来说，SSH 的难点不在“用”，
而在 第一次配置公钥、私钥。

一旦配好，后面反而比 HTTPS 更省心。


二、HTTP / HTTPS 和 SSH 的区别

2.1 HTTPS 方式本质上是什么

HTTPS 也是一种网络通信协议。
它本来主要是给网页用的，比如：打开网站、登录网页、提交表单、下载内容等。
Git 也可以借助 HTTPS 和 GitHub 通信。

所以当用户执行：
git clone https://github.com/用户名/仓库名.git
本质上就是：
Git 通过 HTTPS 去访问 GitHub 这个网站提供的仓库服务。

2.2 HTTPS 和 SSH 最本质的区别

最核心的区别不是“地址长得不一样”，
而是：
它们验证身份的方式不一样。

HTTPS 更像：
“我是这个账号，我来登录一下。”


SSH 更像：
“这台电脑有我认可的钥匙，所以允许它访问。”

2.3 从使用体验上看，两者有什么不同

HTTPS 的体验
优点：
上手直接
不需要先生成密钥
复制仓库地址就能开始
缺点：
第一次 push 往往要登录授权
有时凭证、代理、浏览器跳转会让新手困惑
换电脑后通常还要重新配置认证

SSH 的体验
优点：
配好一次后，后续更顺手
更适合长期开发
多次 push / pull 时体验更稳定

缺点：
第一次配置稍麻烦
要理解公钥、私钥
如果密钥没加对，容易报 Permission denied (publickey) 这类错误

三、使用 SSH 之前要做什么
适用场景：
你想用 SSH 方式连接 GitHub
你不想每次 push 都走浏览器登录授权
你想长期稳定地使用 Git

检查电脑里有没有 SSH 密钥
在 Git Bash / Git CMD 中输入：
cd ~/.ssh
ls

如果看到类似文件：
id_ed25519
id_ed25519.pub
或者：
id_rsa
id_rsa.pub
说明电脑里可能已经有 SSH 密钥了。

含义：
没有 .pub 的文件：私钥
带 .pub 的文件：公钥

注意：
私钥不能发给别人，也不要上传到 GitHub。
GitHub 上添加的是 公钥。


四、生成 SSH 密钥
适用场景：
电脑里还没有 SSH 密钥
或者你想专门为 GitHub 新生成一套密钥

4.1 推荐使用 ed25519
命令：
ssh-keygen -t ed25519 -C "你的GitHub邮箱"
解释：
ssh-keygen：生成 SSH 密钥
-t ed25519：指定密钥类型为 ed25519
-C：添加备注，一般写 GitHub 邮箱，方便识别

4.2 执行后会看到什么
通常会提示：
Enter file in which to save the key (/c/Users/你的用户名/.ssh/id_ed25519):
这里一般直接回车即可。

意思是：
把密钥保存到默认位置。

接着可能提示：
Enter passphrase (empty for no passphrase):
这里是给私钥再加一层密码。

你有两种选择：
直接回车
表示不设置额外密码，使用更省事
输入密码
表示私钥更安全，但以后用的时候可能会多一步验证

4.3 生成成功后会多出什么文件

一般会生成两个文件：
id_ed25519
id_ed25519.pub
含义：
id_ed25519
私钥，不能泄露
id_ed25519.pub
公钥，可以添加到 GitHub

五、把公钥添加到 GitHub
5.1 先查看公钥内容并复制
cat ~/.ssh/id_ed25519.pub

5.2 登录 GitHub 添加 SSH Key
进入 GitHub 后：
点击右上角头像
进入 Settings
找到 SSH and GPG keys
点击 New SSH key

5.3 填写信息
Title：
可以写这台电脑的名字，例如：
My-PC
Key type：
一般选 Authentication Key
Key：
把刚刚复制的公钥完整粘贴进去
然后点击：
Add SSH key

六、测试 SSH 是否配置成功
ssh -T git@github.com

如果第一次连接，可能会提示：
The authenticity of host 'github.com (...)' can't be established.
Are you sure you want to continue connecting (yes/no/[fingerprint])?

输入：
yes

如果成功，通常会看到类似：
Hi 用户名! You've successfully authenticated, but GitHub does not provide shell access.

含义：
你的 SSH 认证成功了
GitHub 识别到你的密钥了
以后可以用 SSH 方式进行 Git 操作了

注意：
这不是报错。
后半句的意思只是：
GitHub 不给你登录成 Linux 终端而已。
但 Git 功能已经可以正常用了。

七、从本地上传现有项目（SSH方式）
适用场景：
本地已经有项目
本地已经 git init 过
本地已经有提交记录
现在想通过 SSH 上传到 GitHub

7.1 链接本地仓库与远程仓库
git remote add origin 远程仓库SSH地址

7.2 检查是否连接成功
git remote -v

7.3 第一次推送到远程仓库
git push -u origin main

第一次推送成功后，后面通常直接用：
git push

7.4 总结
git remote add origin git@github.com:用户名/仓库名.git
git remote -v
git push -u origin main
